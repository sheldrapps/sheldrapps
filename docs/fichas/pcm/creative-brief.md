# PDF Cover Maker (PCM) - Creative Brief

## Purpose

Define the creative, commercial, visual, and localization strategy for Play Store listing assets of PDF Cover Maker (PCM).

This brief guides:

* Play Store copy generation
* feature graphic direction
* screenshot strategy
* regional localization
* golden listings
* bulk ficha generation
* conversion audits
* image generation prompts

This document is the strategic source of truth for PCM fichas and Play Store assets.

## Product Identity

### Working Product Name

PDF Cover Maker (PCM)

### Category

PDF utility / visual organization tool.

### What the App Really Is

A focused tool to create a clean, custom visual cover for an existing PDF file.

In practical product language, “PDF cover” means the visible first page / thumbnail / opening visual of the PDF.

The app helps users make a PDF look better at first glance by choosing an image, adjusting it, previewing the result, and saving a new PDF copy.

### What the App Is Not

* Not a full PDF editor
* Not an OCR tool
* Not a PDF signing tool
* Not a compression, merge, split, or conversion suite
* Not an internal page editing app
* Not an advanced metadata manager
* Not an AI cover generator
* Not a cloud PDF service
* Not a PDF reader or scanner

## Core Positioning

### Main Positioning Statement

Create a better first impression for any PDF.

### Emotional Promise

Make your PDFs look clean, personal, organized, and worth opening.

### Functional Truth

User can:

* pick a PDF
* choose a cover image
* adjust the image
* crop, zoom, rotate, and frame the cover
* preview the result
* save a new PDF copy

### Strategic Balance

PCM positioning priority:

1. Visual result first
2. Speed of workflow second
3. Privacy/local processing as trust proof
4. Safe new-copy output as risk reduction

## Conversion Promise

### Primary Conversion Promise

Create a custom PDF cover in seconds.

### Secondary Conversion Promise

Fast flow from file selection to polished result, without account or uploads.

### Trust Proof

* 100% on-device flow
* no account required
* no cloud upload
* visual adjustment before save
* preview before saving
* output as a new copy

## Product Truth Guardrails

### Safe Claims

Use:

* Make a PDF cover
* Create a PDF cover
* Create a custom PDF cover
* Add a cover to a PDF
* Improve the first visual impression of a PDF
* Make a PDF easier to recognize
* Make PDF thumbnails look cleaner
* Select, adjust, preview, save
* Keep processing on device
* Work without account or signup
* Save a new PDF copy

### Claims to Avoid

Avoid:

* “PDF editor” as the primary claim
* Edit all PDF content
* Edit any PDF page
* OCR promises
* signature or e-sign features
* compression promises
* merge, split, or conversion claims
* scanning claims
* internal page editing claims
* advanced metadata replacement claims
* guaranteed compatibility with every PDF reader
* automatic cover search
* AI-generated covers, unless implemented

### Wording Rule: Critical

Primary brand language must use:

* make a PDF cover
* create a PDF cover
* create a custom PDF cover
* add a cover to a PDF

Do not use “change PDF cover” as the core idea.

“Change PDF cover” can appear only as secondary SEO phrasing when useful, but it should not drive the main positioning.

### PDF Cover Framing Rule

When explaining or visualizing the product, frame “PDF cover” as:

* the visible first page
* the PDF thumbnail
* the opening visual impression
* the cover page users see first

Do not imply hidden metadata cover replacement unless that behavior is explicitly supported.

## Screenshot Strategy

### Recommended Screenshot Order

1. Strong before/after: plain/blank PDF → attractive custom cover
2. Simple flow: Select. Adjust. Done.
3. Visual control: crop, zoom, rotate, frame

1. Strong before/after: plain/blank PDF → attractive custom cover

The first screenshot must sell the outcome, not settings.

### Visual System

Recommended global palette:

* primary background: #151515
* secondary background: #252525
* accent: #B3261E
* optional support accent: #D4574F
* headline color: #FFFFFF
* subline color: #B8B8B8
* bullet/support text color: #E2D2D0

### Accent Rules

Use red as a controlled PDF-associated accent.

Good usage:

* subtle highlight behind the “after” PDF mockup
* before/after markers
* thin lines
* small trust indicators
* controlled depth near the phone or cover result

Avoid:

* full bright red background
* bright red glow behind headline
* aggressive warning-style red
* white panels behind copy
* busy document stacks
* small unreadable UI
* cloud/upload symbols
* AI sparkle effects
* EPUB, ebook, Kindle, or bookstore visuals

## Raw Screenshot Replacement Rules

When using real app screenshots, preserve the raw screenshot composition.

Only replace explicitly allowed elements.

Do not add, remove, invent, or restyle unrelated UI.

### Editor Screen

Allowed:

* Replace only the placeholder/pattern/image area.
* Keep controls, layout, buttons, and app chrome intact.

### Preview Screen

Allowed:

* Replace only the PDF cover preview content.
* Replace PDF name or filename only when the prompt explicitly allows it.

### Saved Result Screen

Allowed:

* Replace only the PDF name and thumbnail image.
* Keep unrelated UI unchanged.

### Safe-Copy Screen

Allowed:

* Replace only the PDF card, PDF name, filename, or output cover content.
* Use a real export/save confirmation screen if available.
* If conceptual, keep it simple: original PDF → new PDF copy.

Reject if:

* the asset invents new UI
* the asset removes app controls
* the asset changes unrelated labels
* the asset adds fake buttons, badges, clouds, or security elements
* the asset modifies the screenshot beyond the allowed scope

## Regional Localization Rules

Each locale should adapt:

* pain point
* examples
* tone
* short description
* screenshot titles
* long description use cases

The visual structure can remain consistent across locales to preserve brand coherence and make bulk asset production easier.

Avoid direct translation clones.

A good localized ficha should feel native in:

* search terms
* document examples
* level of formality
* emotional angle
* headline phrasing

## Recommended Regional Use Cases

Use cases should vary by locale when helpful.

Safe global use cases:

* study files
* manuals
* reports
* catalogs
* notes
* guides
* worksheets
* portfolios
* work documents
* personal documents
* client files
* presentation documents

Avoid using “ebooks” as a global use case because it can confuse PCM with EPUB/Kindle/ECC positioning.

Use “PDF ebooks” only if a specific locale benefits from it and the wording remains clearly PDF-focused.

## Final Creative Direction

### Tone

Clean, practical, modern, confidence-building.

### Visual Mood

Dark, matte, PDF-native, controlled, useful, and premium without feeling corporate-heavy.

### Core Narrative

Not:
“Edit everything in your PDF.”

Yes:
“Create a polished custom cover that improves how your PDF looks at first glance.”

### Quality Bar

Work is acceptable only if:

* copy and assets lead with “make/create a PDF cover”
* no unsupported all-in-one PDF claims appear
* screenshot 1 communicates immediate transformation
* feature graphic communicates PDF cover creation instantly
* privacy and no-upload trust proof is explicit
* save-as-copy behavior is present or clearly explained
* localized angles feel native, not direct translation clones
* visuals avoid EPUB, Kindle, ebook, cloud, AI, OCR, scanner, converter, and full-editor cues

## Final Approval Rule

Approve a PCM ficha or asset only if it clearly communicates:

“I can create a better-looking PDF cover from my own image, adjust it, preview it, and save a new copy without uploading my file.”
