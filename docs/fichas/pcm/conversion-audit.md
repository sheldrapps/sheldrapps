# PDF Cover Maker - Conversion Audit

## Purpose

Evaluate whether the generated Play Store listings for PDF Cover Maker are strong enough for conversion, accurate to the product, visually clear, and safe from unsupported claims.

This audit should be used after generating each locale ficha and before using the listing in Play Console or asset production.

## Product Conversion Goal

PDF Cover Maker should convert by making users immediately understand:

“I have a PDF with an ugly, blank, generic, plain, or unclear first page. This app lets me create a better PDF cover quickly, privately, and without uploading my file.”

The listing should sell the visual result first, then explain the workflow, privacy proof, and safe save-as-copy behavior.

## Core Product Framing

For PDF Cover Maker, “PDF cover” should be framed as the visible first page / PDF thumbnail / opening page of the PDF.

Do not imply hidden metadata cover replacement unless the product truly supports that behavior.

Safe framing:

* Create a better PDF cover
* Add a cover to a PDF
* Replace a plain first page with a better cover
* Improve the PDF thumbnail / first-page preview
* Save a new PDF copy with the updated cover

Avoid framing that implies unsupported internals:

* Change hidden PDF cover metadata
* Fix every PDF preview everywhere
* Guarantee all PDF readers will display the cover the same way
* Edit any page of the PDF
* Fully redesign the PDF document

## Passing Threshold

Minimum acceptable score:
80 / 100

Strong score:
90 / 100 or higher

Reject or revise immediately if:

* Screenshot 1 does not show a before/after PDF cover transformation.
* The copy implies full PDF editing.
* The listing does not mention local processing, no uploads, or equivalent privacy proof.
* The feature graphic does not communicate PDF cover creation in one glance.
* The copy implies guaranteed compatibility with every PDF or every PDF reader.
* The listing frames the product as a PDF converter, scanner, compressor, OCR tool, or AI cover generator.

## Long Description Audit

### Expected Structure

A strong long description should include:

1. Opening pain:
   A PDF looks plain, ugly, generic, unclear, or hard to recognize.

2. Product promise:
   Create a better-looking cover for your PDF.

3. Workflow:
   Select PDF, choose image, adjust, preview, save a new copy.

4. Use cases:
   Study files, manuals, reports, catalogs, notes, guides, worksheets, portfolios, work documents, personal documents, client files, or presentation documents.

5. Privacy:
   On-device processing, no account, no uploads, no cloud processing.

6. Limitation / positioning:
   Focused on PDF cover creation, not a full PDF editor.

7. Closing motivation:
   Make PDFs cleaner, easier to recognize, and better organized.

### Score Target

9–10 / 10

### Recommended Length

A strong long description will usually be around 700–1300 characters.

Do not reject only because it is under 1000 characters. Reject if it feels shallow, feature-only, unclear, or underdeveloped.

### Reject If

* It promises editing all PDF pages.
* It says OCR, compression, merge, split, signatures, forms, scanning, or conversion.
* It says “works with every PDF” as an absolute guarantee.
* It reads like a feature list only.
* It does not explain the user pain.
* It does not explain privacy.
* It does not explain save-as-copy behavior.
* It makes the app sound like a general PDF suite.

## Raw Screenshot Replacement Rules

When a real app screenshot is used, preserve the raw screenshot composition.

Only replace explicitly allowed elements.

Do not add, remove, invent, or restyle unrelated UI.

### Allowed Replacement Examples

Editor screen:

* Replace only the placeholder/pattern/image area.
* Keep controls, layout, buttons, and app chrome intact.

Preview screen:

* Replace only the PDF cover preview content.
* Replace PDF name/filename only if the prompt explicitly allows it.

Saved result screen:

* Replace only the PDF name and thumbnail image.
* Keep all unrelated UI unchanged.

Safe-copy screen:

* Replace only the PDF card, PDF name, filename, or output cover content.
* Do not create fake success UI unless the screen already supports it or the prompt explicitly requests a conceptual composition.

### Reject If

* The generated asset invents new UI.
* The generated asset removes existing app controls.
* The generated asset changes unrelated labels.
* The generated asset adds fake buttons, fake badges, fake security elements, or fake clouds.
* The generated asset modifies the real screenshot beyond the allowed scope.

## Copyright and Privacy Audit

### Required

Visuals must avoid:

* copyrighted real covers
* recognizable brand documents
* real personal data
* real invoices
* real IDs
* real bank statements
* private addresses
* sensitive documents
* tiny unreadable legal text pretending to be real

### Safe Visual Content

Use:

* generic public-domain style document concepts
* neutral sample PDFs
* abstract report covers
* study guide covers
* manual-like covers
* catalog-like covers
* portfolio-like covers
* generic nature/photo/texture cover images
* fake names and filenames that are clearly generic

### Reject If

* A real copyrighted book, brand, invoice, or private document appears.
* The content looks like a real confidential document.
* The PDF cover contains readable personal data.
* The design implies official affiliation with Adobe, Google, Microsoft, Kindle, or any document platform.

## Final Review Checklist

Before approving a ficha, confirm:

* App name is under 30 characters.
* Short description is clear and within limit.
* Long description is persuasive and not shallow.
* Long description explains pain, workflow, privacy, and safe copy.
* Screenshot 1 shows the main before/after transformation.
* Feature graphic communicates PDF cover creation instantly.
* The app does not look like a full PDF editor.
* The app does not look like a PDF scanner, converter, compressor, or OCR tool.
* Privacy is visible and believable.
* Save-as-copy behavior is visible or clearly explained.
* Claims match actual app behavior.
* Locale strategy is not a direct translation clone.
* Regional examples feel natural.
* Red accents are controlled and readable.
* No copyrighted real covers or personal data appear in visuals.
* All screenshots have clear conversion intent.
* Raw screenshot replacement rules are respected.
* “PDF cover” is framed as first page / thumbnail / visible cover, not unsupported hidden metadata.

## Approval Decision

Approved:
Only if total score is 80/100 or higher and no critical rejection rule is triggered.

Strong:
90/100 or higher, with no major visual/copy issue.

Needs revision:
65–79/100, or one major visual/copy issue exists.

Rejected:
Below 65/100, or any critical rejection rule is triggered.

Critical rejection triggers:

* Screenshot 1 fails to show the transformation.
* Privacy/local processing is missing.
* Unsupported PDF editing claims are present.
* Feature graphic does not communicate PDF cover creation.
* The app looks like a full PDF editor, scanner, converter, compressor, OCR tool, or AI generator.
* The asset uses cloud/upload visuals.
* The asset uses copyrighted covers, real private documents, or personal data.
