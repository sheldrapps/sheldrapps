# PDF Cover Maker - Conversion Audit

## Purpose

Evaluate whether the generated Play Store listings for PDF Cover Maker are strong enough for conversion, accurate to the product, and safe from unsupported claims.

This audit should be used after generating each locale ficha and before using the listing in Play Console or asset production.

## Product Conversion Goal

PDF Cover Maker should convert by making users immediately understand:

“I have a PDF with an ugly, blank, generic, or unclear cover. This app lets me create a better one quickly, privately, and without uploading my file.”

The listing should sell the visual result first, then explain the workflow and privacy proof.

## Global Audit Criteria

| Area | Score | What Good Looks Like | Common Failure |
|---|---:|---|---|
| App name clarity | /10 | Name clearly communicates PDF cover creation and stays under 30 characters | Name sounds like a full PDF editor or exceeds limit |
| Short description strength | /10 | Clear, searchable, and conversion-focused | Too generic, too technical, or not PDF-specific |
| Long description depth | /10 | Explains pain, outcome, workflow, privacy, use cases, and limits | Thin summary under 1000 characters |
| Screenshot 1 hook | /10 | Shows clear before/after PDF cover transformation | Starts with settings, file picker, permissions, or editor UI |
| Feature graphic clarity | /10 | Communicates “better PDF cover” in one glance | Looks like a generic PDF tool |
| Regional differentiation | /10 | Locale has a distinct angle, examples, and visual tone | Same ficha translated mechanically |
| Trust proof clarity | /10 | Local processing, no account, no uploads are visible and believable | Privacy mentioned vaguely or only at the end |
| Workflow clarity | /10 | User understands select → adjust → preview → save | Flow feels complex or hidden |
| Unsupported claim risk | /10 | Avoids OCR, compression, merge, split, signatures, full editing | Implies full PDF editing or universal compatibility |
| Visual specificity | /10 | Assets describe concrete scenes, layout, colors, safe copy areas | Generic “show app screen” instructions |

## Passing Threshold

Minimum acceptable score:
80 / 100

Strong score:
90 / 100 or higher

Reject or revise immediately if:
- Screenshot 1 does not show a before/after transformation.
- The copy implies full PDF editing.
- The listing does not mention local processing / no uploads.
- The feature graphic does not communicate PDF cover creation in one glance.
- The long description is under 1000 characters without a strong reason.

## App Name Audit

### Expected

PDF Cover Maker

### Score

10 / 10

### Reasoning

The name is short, clear, searchable, and focused. It contains the strongest keywords: PDF, Cover, Maker. It does not overpromise full PDF editing.

### Risk

Low.

### Recommendation

Keep the app name as PDF Cover Maker across locales unless a localized name clearly improves ASO without exceeding 30 characters.

## Short Description Audit

### Expected quality

The short description should include:
- PDF
- cover / portada / localized equivalent
- create / make / crear
- speed or simplicity when possible

### Strong examples

en-US:
Create custom PDF covers in seconds

es-MX:
Crea portadas para PDF en segundos

pt-BR:
Crie capas para PDF em segundos

de-DE:
PDF-Cover schnell erstellen

### Score target

8–10 / 10

### Common issues

Reject if:
- It says only “Edit PDFs easily”.
- It sounds like a full PDF editor.
- It does not mention PDF.
- It uses vague wording like “Improve your files”.
- It exceeds Play Store short description limits.

## Long Description Audit

### Expected structure

A strong long description should include:

1. Opening pain:
“Not happy with your PDF cover?”

2. Product promise:
Create a better-looking cover for your PDF.

3. Workflow:
Select PDF, choose image, adjust, preview, save copy.

4. Use cases:
ebooks, study files, manuals, reports, catalogs, notes, guides, personal documents.

5. Privacy:
100% on device, no account, no uploads, no cloud processing.

6. Limitation / positioning:
Not a full PDF editor.

7. Closing motivation:
Make PDFs cleaner, easier to recognize, and better organized.

### Score target

9–10 / 10

### Reject if

- It promises editing all PDF pages.
- It says OCR, compression, merge, split, signatures, forms, or conversion.
- It says “works with every PDF” as an absolute guarantee.
- It is below 1000 characters.
- It reads like a feature list only.

## Feature Graphic Audit

### Expected

The feature graphic must communicate the product in one glance.

### Required elements

- Clear title about creating PDF covers.
- Before/after visual.
- Red PDF-associated accent, but not aggressive.
- Safe dark copy area.
- No clutter.
- No tiny unreadable UI.
- At least one trust cue or workflow cue.

### Strong title examples

CREATE BETTER PDF COVERS  
CREA PORTADAS PARA PDF  
CRIE CAPAS PARA PDF

### Score target

9–10 / 10

### Reject if

- It looks like a PDF reader.
- It looks like a full PDF editor.
- It uses a full bright red background that hurts readability.
- The before/after transformation is not obvious.
- It shows cloud icons or implies upload.

## Trust Proof Audit

### Required trust claims

At least one of these must appear in high-visibility copy:

- 100% on your device
- No accounts
- No uploads
- No cloud processing
- Save a new copy

### Score target

9–10 / 10

### Reject if

- Privacy appears only in the notes.
- The app seems to upload PDFs.
- The listing uses cloud-related visuals.

## Unsupported Claim Risk Audit

### Safe claims

Allowed:
- Create PDF covers
- Change/improve PDF cover
- Choose image
- Crop, zoom, rotate, adjust
- Preview result
- Save a new copy
- Local processing
- No account
- No upload
- Useful for study files, manuals, ebooks, reports, catalogs, guides, personal PDFs

### Risky or forbidden unless implemented

Avoid:
- Full PDF editor
- Edit all pages
- OCR
- Compress PDF
- Merge PDF
- Split PDF
- Sign PDF
- Fill PDF forms
- Convert PDF
- Repair corrupted PDFs
- Guaranteed to work with every PDF
- Cloud backup
- AI generation

### Scoring

10 / 10:
No unsupported claims.

7 / 10:
Minor vague phrase that should be clarified.

5 / 10 or lower:
Implies full PDF editing or unsupported PDF operations.

## Visual Specificity Audit

### Expected

Every feature graphic and screenshot must define:

- title
- subline
- headline color
- subline color
- wrapper
- background dimension
- primary background
- secondary background
- accent color
- accent placement
- safe copy area
- allowed texture
- forbidden elements
- image direction
- conversion intent

### Reject if

- Asset prompt says only “show app screen”.
- No safe copy area is defined.
- No background rules are defined.
- Visuals are generic or impossible to produce consistently.

## Final Review Checklist

Before approving a ficha, confirm:

- App name is under 30 characters.
- Short description is clear and within limit.
- Long description is persuasive and not shallow.
- Screenshot 1 shows the main before/after transformation.
- Feature graphic communicates PDF cover creation instantly.
- The app does not look like a full PDF editor.
- Privacy is visible and believable.
- Claims match actual app behavior.
- Locale strategy is not a direct translation clone.
- Red accents are controlled and readable.
- No copyrighted real covers or personal data appear in visuals.
- All screenshots have clear conversion intent.

Only if total score is 80/90 or higher and no critical rejection rule is triggered.

Needs revision:
If score is 65–79 or one major visual/copy issue exists.

If score is below 65, screenshot 1 fails, privacy is missing, or unsupported PDF editing claims are present.
