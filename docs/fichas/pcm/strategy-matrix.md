# PDF Cover Maker - Strategy Matrix

## Purpose

Define the regional conversion strategy for PDF Cover Maker across all supported locales.

The app should be positioned as a focused, private, on-device tool for creating better PDF covers. The core promise is not full PDF editing, but making PDFs look cleaner, easier to recognize, and more intentional at first glance.

The strategy follows conversion-first logic:

1. Sell the visual outcome first.
2. Show the workflow second.
3. Support the promise with privacy and safe-copy trust proof.

This matrix should guide:

* localized Play Store fichas
* short descriptions
* long descriptions
* feature graphics
* screenshot titles
* screenshot visual direction
* bulk asset generation
* conversion audits

## Global Positioning

### Main Promise

Create a better cover for your PDF in seconds.

### Functional Truth

The user selects a PDF, chooses an image, adjusts the layout, previews the result, and saves a new copy.

### Product Framing

For PCM, “PDF cover” means the visible first page / thumbnail / opening visual impression of the PDF.

Do not imply hidden metadata cover replacement unless explicitly supported.

### Trust Proof

100% on device.
No accounts.
No uploads.
Save a new copy.

### Primary Conversion Hook

Turn an ugly, blank, generic, plain, or unclear PDF first page into something clean, attractive, and recognizable.

### Main Visual System

Dark premium background, restrained PDF-red accents, clear before/after transformation, readable text, and no overloaded technical UI.

Recommended global palette:

* primary background: #151515
* secondary background: #252525
* accent: #B3261E
* optional support accent: #D4574F
* headline color: #FFFFFF
* subline color: #B8B8B8
* bullet/support text color: #E2D2D0

## Screenshot Strategy by Locale

### Global Screenshot Order

Recommended 6-screenshot order:

1. Main transformation / before-after
2. Simple workflow
3. Visual control
4. Preview before saving
5. Privacy / on-device processing
6. Save new copy or organization outcome

Recommended 5-screenshot order:

1. Main transformation / before-after
2. Simple workflow
3. Visual control and preview
4. Privacy / on-device processing
5. Cleaner PDF files / organization outcome / save-copy proof

### Screenshot 1: Main Transformation

Goal:
Sell the visual outcome immediately.

Expected:
Plain, blank, generic, scanned, or unclear PDF first page → attractive custom PDF cover.

Reject if:
The screenshot starts with settings, file picker, permissions, editor controls, or privacy-only text.

### Screenshot 2: Simple Workflow

Goal:
Show that the app is fast and easy.

Expected:
Select PDF → choose image → adjust → done.

Common copy pattern:
Select. Adjust. Done.

### Screenshot 3: Visual Control

Goal:
Show that the user controls the final cover.

Expected:
Crop, zoom, rotate, align, frame.

The screenshot should not look like a full PDF editor or complex design suite.

### Screenshot 4: Preview Before Saving

Goal:
Reassure the user that the result can be checked before export.

Expected:
Final PDF cover preview is centered, polished, and clearly better than the original.

### Screenshot 5: Privacy Trust

Goal:
Build confidence.

Expected:
100% on your device.
No accounts.
No uploads.

Avoid:
Cloud icons, fake security badges, login/account screens, or cyber-security-style visuals.

### Screenshot 6: Safe Copy or Organization Outcome

Goal:
Reduce risk or show long-term value.

Expected options:

* Original PDF → new PDF copy with updated cover
* Saved result screen
* Cleaner PDF list with better thumbnails
* Local file result with updated cover

Do not force a library screen if the app does not have a real library/list view.

## Global Feature Graphic Direction

### Title Pattern

Create better PDF covers / Crear portadas PDF / localized equivalent.

### Subline Pattern

Choose an image, adjust it, and save / localized equivalent.

### Required Visual

A before/after PDF transformation, not just the app UI.

Before:
Plain, blank, generic, or unclear PDF first page.

After:
Polished custom PDF cover.

### Required Trust Cue

At least one of:

* 100% on your device
* No uploads
* Preview before saving
* Save a new copy

### Avoid

* full red background
* bright red glow behind headline
* tiny UI screenshots
* cloud icons
* too many bullets
* EPUB or Kindle visuals
* AI sparkle effects
* any visual that implies full PDF editing

## Global Copy Guardrails

### Safe Claims

Use:

* create a cover for your PDF
* make a PDF cover
* create a custom PDF cover
* add a cover to a PDF
* choose an image
* crop, zoom, rotate, adjust
* preview before saving
* save a new copy
* process locally
* no account
* no upload
* useful for study files, manuals, reports, catalogs, guides, notes, worksheets, portfolios, work documents, and personal PDFs

### Secondary SEO Phrasing

Allowed only when useful and not used as the main positioning:

* change PDF cover
* replace PDF cover
* edit PDF cover

### Avoid Claims

Avoid:

* edit any PDF
* edit all pages
* full PDF editor
* OCR
* compress PDF
* merge PDF
* split PDF
* sign PDF
* fill forms
* scan documents
* convert PDF
* repair damaged PDFs
* guaranteed compatibility with every PDF
* guaranteed identical preview in every PDF reader
* cloud backup
* AI generation
* automatic cover search
* ready-made templates, unless implemented

## Visual Differentiation Rules

Each locale can reuse the same product truth and visual system, but should not feel like a direct translation clone.

Recommended variation by region:

* US: polished productivity and digital organization.
* Mexico / Brazil: practical, warm, personal documents and study PDFs.
* Germany / Japan: precise, minimal, privacy-heavy.
* France / Italy: refined, visual presentation.
* Saudi Arabia: elegant, privacy-first, high contrast.
* India: study/work practical utility.
* Korea / China / Taiwan: clean digital organization and fast workflow.
* Russia: direct utility and local control.

Differentiate primarily through:

* title wording
* subline wording
* document examples
* filenames
* local use cases
* tone
* cover imagery style

Do not over-differentiate the visual system unless there is a strong conversion reason.

## Quality Bar

A generated locale ficha is acceptable only if:

* Screenshot 1 shows a clear before/after transformation.
* The first impression does not look like a full PDF editor.
* The feature graphic communicates “better PDF cover” in one glance.
* Privacy is explicit but not the only selling point.
* The long description does not overpromise unsupported PDF features.
* The copy leads with create/make a PDF cover.
* “Change PDF cover” is not the main positioning.
* The app is framed as improving the visible first page / thumbnail / opening visual.
* The copy feels local, not mechanically translated.
* No cloud, AI, OCR, scanner, converter, EPUB, Kindle, or full-editor cues appear.
